### 1.3 빌드 및 배포 절차

#### EC2 서버 초기 세팅 (최초 1회)

```bash
# 1. Docker 네트워크 생성
docker network create gguljob-network
docker network create gguljob-network-dev

# 2. .env 파일 작성
vi .env        # 개발 환경용
vi .env-prod   # 운영 환경용
# JWT_SECRET 생성: openssl rand -hex 32

# 3. SSL 인증서 발급 (Certbot)
sudo certbot certonly --standalone -d j14e107.p.ssafy.io

# 4. 운영 환경 전체 기동
docker compose --env-file .env-prod -f docker-compose.prod.yml up -d

# 5. 개발 환경 전체 기동
docker compose --env-file .env -f docker-compose.dev.yml up -d
```

#### 백엔드 빌드

```bash
cd backend

# Gradle 빌드 (테스트 제외)
chmod +x gradlew
./gradlew clean bootJar -x test --no-daemon

# Docker 이미지 빌드 (수동 시)
docker build -t gguljob-backend:latest .
```

**Dockerfile 멀티 스테이지 빌드 구조:**

| 단계 | 베이스 이미지 | 역할 |
|------|-------------|------|
| Stage 1 (builder) | `eclipse-temurin:21-jdk-jammy` | Gradle 의존성 캐시 레이어 분리 → `./gradlew clean bootJar -x test` 실행 |
| Stage 2 (runtime) | `eclipse-temurin:21-jre-jammy` | 빌드된 JAR만 복사, `curl` 설치 (헬스체크용), `java -jar -Duser.timezone=Asia/Seoul app.jar` 실행 |

#### 프론트엔드 빌드

```bash
cd frontend

# 로컬 빌드
npm ci

# Docker 이미지 빌드 (수동 시)
docker build -t gguljob-frontend:latest .
```

**Dockerfile 멀티 스테이지 빌드 구조:**

| 단계 | 베이스 이미지 | 역할 |

| Stage 1 (builder) | `node:20-alpine` | `npm ci` → `npm run build` (Vite 빌드 → `dist/`) |
| Stage 2 (runtime) | `nginx:alpine` | `dist/` → `/usr/share/nginx/html`, SPA 라우팅용 `default.conf` 적용, 포트 80 노출 |

**백엔드 CI/CD 파이프라인 흐름:**

```
1. Checkout (소스 코드 가져오기)
2. Build Backend (./gradlew clean build -x test)
3. Deploy Blue/Green
   ├── Jenkins Credential에서 .env 파일 복사
   ├── deploy.sh (Dev) 또는 deploy-prod.sh (Prod) 실행
   │   ├── 현재 활성 컬러(Blue/Green) 감지
   │   ├── 비활성 컨테이너 제거 (좀비 방지)
   │   ├── 새 컨테이너 빌드 & 기동 (docker compose up -d --build --no-deps)
   │   ├── Health Check (GET /api/v1/health, 최대 75초 대기)
   │   ├── Nginx service-url.inc 업데이트 (docker exec tee)
   │   ├── nginx -t (설정 검증)
   │   ├── nginx -s reload (무중단 트래픽 전환)
   │   └── 이전 컨테이너 중지 & 제거
   └── .env 파일 삭제 (보안)
4. Cleanup (docker image prune -f)
5. Mattermost 알림 (성공/실패)
```

```
1. Checkout
2. docker compose up -d --build frontend-dev (Dev) / frontend (Prod)
3. Cleanup (docker image prune -f)
4. Mattermost 알림
```

### 1.4 배포 시 특이사항

#### 1) Blue/Green 무중단 배포

- 백엔드는 **Blue**와 **Green** 두 개의 컨테이너를 교대 운용합니다.
- `nginx/conf.d/service-url.inc` (Prod) / `nginx/conf.d-dev/service-url.inc` (Dev) 파일이 현재 활성 컨테이너 주소를 저장합니다.
- 배포 시 `docker exec tee`로 직접 Nginx 컨테이너 내부 파일을 갱신 후 `nginx -s reload`로 트래픽을 전환합니다.
- 헬스체크 엔드포인트: `GET /api/v1/health` → HTTP 2xx 응답 필요
- 헬스체크 실패 시 (10회 재시도, 총 ~75초) 자동으로 배포를 중단하고 이전 컨테이너를 유지합니다.

#### 2) Vite 빌드 시 환경변수 주입 주의

- `VITE_` 접두사 환경변수는 **빌드 시점에 번들에 정적으로 삽입**됩니다.
- `docker build` 실행 전에 `frontend/.env` 파일이 반드시 존재해야 합니다.
- 런타임에는 변경할 수 없으므로, 환경변수가 바뀌면 **반드시 재빌드** 필요합니다.

#### 3) 운영 환경 보안 설정

- **Swagger UI 비활성화**: `application-prod.yml`에서 `swagger-ui.enabled: false`, `api-docs.enabled: false`
- **DB/Redis 포트 미노출**: 운영 환경의 MySQL, Redis는 `127.0.0.1` 바인딩 또는 Docker 내부 네트워크만 사용
- **로그 관리**: Docker JSON 파일 드라이버, 파일당 최대 10MB, 최대 3~5개 파일 유지
- **Portainer 접근 제어**: Nginx Basic Auth (`.htpasswd`) + Portainer 자체 인증 이중 보호

#### 4) JPA DDL 정책

- **Dev / Prod 모두**: `ddl-auto: validate` (스키마 검증만, 자동 변경 없음)
- 스키마 변경 시 별도 마이그레이션 SQL을 수동 실행해야 합니다.

#### 5) SSL 인증서 갱신

- Let's Encrypt 인증서 사용 (유효기간 90일)
- 인증서 경로: `/etc/letsencrypt/live/j14e107.p.ssafy.io/`
- Nginx 컨테이너에 호스트의 `/etc/letsencrypt`를 볼륨 마운트하여 사용
- 갱신: `sudo certbot renew` → Nginx 컨테이너 재시작 또는 `nginx -s reload`

#### 6) Neo4j 초기 설정

- 최초 기동 시 `NEO4J_AUTH=neo4j/${NEO4J_PASSWORD}` 환경변수로 비밀번호 자동 설정
- Dev: heap 512m~1G / Prod: heap 1G~2G

#### 7) Jenkins 컨테이너 특이사항

- Jenkins 컨테이너 내부에 Docker CLI + Compose Plugin이 설치되어 있어 호스트 Docker 소켓(`/var/run/docker.sock`)을 통해 컨테이너를 직접 제어합니다.
- `DOCKER_GID` 빌드 인자를 호스트의 docker 그룹 GID와 맞춰야 소켓 권한이 정상 작동합니다.
  - 확인 방법: `stat -c '%g' /var/run/docker.sock` (보통 988 또는 999)
- Jenkins 메모리 제한: 10GB (`mem_limit: 10g`)

#### 8) 운영 Prod 빌드 시 버전 검증

- `Jenkinsfile-be-prod`에서 `build.gradle`의 `version`이 `SNAPSHOT`을 포함하면 빌드가 자동 실패합니다.
- 운영 배포 전 반드시 정식 버전 (예: `1.0.0`)으로 변경해야 합니다.

